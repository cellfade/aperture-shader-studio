# Aperture — Filtered Video Export: QA Punch List

**Build:** PASS — typecheck and production build both clean. `rm -rf .next && npx tsc --noEmit` exited 0 (no type errors); `npx next build` exited 0 (Next.js 16.2.9 / Turbopack, compiled ~1.2s, TS check passed, all 5 static pages generated). No source edits made.

Merged and deduplicated from Code, A11y, and Design QA passes. Grouped by priority. Each item: **title** — detail — fix — file.

> **Resolution (fix pass):** **P0 fixed** — export controls are now persistent toggle buttons (Export↔Cancel on the same node), so focus is never lost on completion; this also gave each Cancel a distinct `aria-label`. **High-value P1s fixed:** primary/secondary hierarchy + noun-first labels (`Frames · zip` / `Video · mp4`, dropped the redundant "EXPORT") so the two actions are instantly distinguishable; capability-fallback line now `role="status" aria-live="polite"` at `text-foreground/70` contrast; **frame-source abort is now cooperative** (settle-once guards, quiesces the consumer before teardown, no double-close/double-reject) and has **decoder backpressure** (bounds in-flight decode + pending frames to ≤24 — long clips no longer risk frame-pool exhaustion). E2E re-verified: 300-frame (10s) export downloads a valid MP4, mid-run Cancel is clean (0 console errors, focus preserved), and cancel→rerun works. The throwaway `/spike` route was removed (spike findings kept in `docs/spike-webcodecs.md`).
> **Deferred (documented P2 fast-follows, not ship-blocking):** per-frame PNG-roundtrip perf + the content-presence readback gate (can rarely mis-pass a stale/duplicate frame — the one transient failure seen in testing; switch to a change-detecting signature + reuse one canvas); VFR-source timestamp/duration normalization; `isSupported` on the exact (clamped/evened) export dims; `RenderCore.create` abort/timeout; eager muxer-buffer release in `dispose()`; distinct "export module failed to load" message for the dynamic-import path; `aria-describedby` precondition hints on disabled export buttons (currently conveyed via `title`).

## P0 — Must fix before ship

- [ ] **Focus is lost when Cancel/Export button unmounts on run completion or cancellation** — In both the sequence block (lines 533-562) and MP4 block (lines 566-595), the UI is a hard ternary on `running`/`vidRunning`. While running, the only control is `Cancel`; when the run ends, is cancelled, or errors, `setRunning(false)`/`setVidRunning(false)` swaps Cancel out and mounts the `Export …` button. A keyboard/SR user focused on the unmounted node loses focus to `<body>`, losing their place; next Tab restarts from the top of the document. — **Fix:** Move focus deterministically on the `true→false` edge: give the Export button a ref and refocus it in a `useEffect` keyed on `running`/`vidRunning`, tracking the prior value with a ref and guarding against stealing focus if the user moved on. Alternatively use a single persistent button whose label/handler switches between Export and Cancel so the DOM node never unmounts (also resolves the duplicate-Cancel naming issue below). — `components/studio/video-stage.tsx`

## P1 — High priority

- [ ] **Abort during decode leaves decoder + in-flight consumer chain running detached; can double-reject/double-close** — `onAbort()` (line 173) and the main async IIFE (line 247) race on the same Promise. `onAbort` closes the decoder and rejects, but the IIFE's `await decoder.flush()` / `await consumerChain` keep running: `flush()` on a closed decoder throws into the IIFE catch, which calls `decoder.close()` again and re-rejects. More importantly, any frame inside `await onFrame(decoded)` is owned by the consumer chain, which `onAbort` does NOT stop — the chain keeps mutating state / closing frames after the outer promise rejected, while the consumer may already be disposing the RenderCore mid-`render()`. Closing the decoder mid-decode with chunks queued drops the frame pool. — **Fix:** Make abort cooperative: set an `aborted` flag in `onAbort` and signal the consumer instead of close+reject. Gate IIFE steps on it (skip `flush()` if `decoder.state==="closed"` or aborted; bail before `await consumerChain` if aborted). Await `consumerChain` to quiesce BEFORE closing decoder/encoder/core so no `onFrame` is mid-render at teardown. Guard reject with a `settled` boolean so `onAbort` and the catch can't both reject. — `lib/studio/video-export/frame-source.ts:172-289`

- [ ] **No decoder backpressure — out-of-range frames dropped by CTS but consumer can outrun the window and exhaust the frame pool** — `onSamples` decodes EVERY sample with `cts<=outUs` in a tight synchronous for-loop with no await; `decoder.decodeQueueSize` is never checked. A 30s range at 30-60fps queues 900-1800 `EncodedVideoChunk`s plus decoded `VideoFrame`s into the small frame pool before any backpressure. `drainPending(false)` only gates the CONSUMER (via `consumerChain`), not the decoder, and never awaits — `onSamples` returns immediately and mp4box keeps feeding. The encoder side has explicit `queueSize` backpressure (encode-filtered-video.ts:252) but the decode side does not, so a long clip can exhaust the `VideoFrame` pool and stall/error the decoder before any frame is consumed. — **Fix:** Apply real decoder backpressure: after `decoder.decode(chunk)`, pause feeding when `decodeQueueSize` exceeds a threshold (~16-32). Since `onSamples` is a sync mp4box callback, restructure to pull samples via `setExtractionOptions` with `nbSamples` batching, or buffer samples and drive decode from an async loop that awaits both `decodeQueueSize` and `consumerChain`. At minimum, bound `pending` so a slow consumer can't accumulate unbounded decoded frames. — `lib/studio/video-export/frame-source.ts:221-244`

- [ ] **Capability-fallback line ("Needs Chrome/Edge/Safari") is not announced and uses low-contrast muted text** — The fallback (lines 598-602) renders only after the async `VideoEncoder.isConfigSupported` probe (lines 118-137) resolves `videoExportOk=false`. It is a plain `<p class="text-muted-foreground">` with no `role`/`aria-live`: (1) an SR user can have the "Export MP4" button silently never appear with no spoken explanation; (2) `text-muted-foreground` on the dark panel is the lowest-contrast token and risks failing WCAG 1.4.3 (4.5:1). It also carries a stray `tabular-nums` class meaningless on prose. — **Fix:** Wrap the message in a polite live region (`role="status"` / `aria-live="polite"`) so it announces when the probe resolves; bump from muted-foreground to a foreground/80-level token for contrast; drop `tabular-nums`. Optionally associate it with the export control via `aria-describedby`. — `components/studio/video-stage.tsx:598-602` (probe 118-137)

- [ ] **Disabled Export buttons give no reason and are inconsistent with the aria-disabled Capture button** — "Export N frames" (553-560) and "Export MP4" (586-593) use the real `disabled` attribute when `!seqValid`/`!vidValid` or while the other export runs. Real-disabled buttons leave the tab order and expose no reason, so a keyboard/SR user can't discover why export is unavailable (out-point not after in-point, or "one export at a time"). Meanwhile the Capture button (465-476) deliberately uses `aria-disabled` + `aria-busy` + `onClickCapture` to stay focusable — inconsistent patterns within one component. — **Fix:** Standardize on the focusable `aria-disabled` pattern used by Capture, or keep `disabled` but add `aria-describedby` pointing to a short hint stating the precondition ("Set an out-point after the in-point" / "Another export is running"). Ensure the reason is perceivable to SRs, not only conveyed by greyed appearance. — `components/studio/video-stage.tsx:553-560, 586-593`

- [ ] **Two "Cancel" buttons share an identical accessible name with no distinguishing context** — The sequence Cancel (538-544) and MP4 Cancel (571-577) both expose the bare name "Cancel". Although only one export runs at a time (mutual lock), the buttons are structurally independent and an SR rotor shows two indistinguishable controls. No `aria-busy`/association ties Cancel to the in-progress status it controls. — **Fix:** Give each a distinct `aria-label` ("Cancel frame export" / "Cancel MP4 export"). Adopting the single-persistent-button approach from the P0 focus finding resolves this automatically. — `components/studio/video-stage.tsx:538-544, 571-577`

- [ ] **Two co-equal export buttons read as one ambiguous pair (no primary action)** — EXPORT 8 FRAMES and EXPORT MP4 sit side-by-side in the same flex group (line 531) with identical styling (same `bg-foreground/[0.06]` fill, same `hover:bg-foreground/15`, same padding, same mono-caps type at 557 and 590). Nothing signals they produce fundamentally different artifacts (a zip of stills vs. a single video). Equal weight + only `gap-2` forces the user to read both labels; no implied primary. This is the core clutter source. — **Fix:** Establish one primary, one secondary. MP4 is the headline deliverable for a video stage — give EXPORT MP4 the filled treatment and demote EXPORT 8 FRAMES to the quiet outline-only variant used by Set in/Set out (border only, `hover:bg-foreground/10`). Keep identity monochrome; differentiate via fill weight, not color. Add a hairline divider or bump to `gap-3` so they don't read as a segmented control. — `components/studio/video-stage.tsx:531, 557, 590`

- [ ] **"EXPORT" prefix repeated on both buttons wastes the most-scanned word** — Both CTAs start with EXPORT (EXPORT 8 FRAMES / EXPORT MP4). In uppercase mono with 0.1em tracking the shared leading word makes them look near-identical, pushing the distinguishing tokens (FRAMES vs MP4) to the far right where they're easiest to miss. The dynamic frame count also makes the left button wider, creating a ragged, unbalanced pair. — **Fix:** Drop the redundant verb now that both live under an explicit Sequence/export context. Label by output: left = "FRAMES .ZIP" (or "8 FRAMES (ZIP)"), right = "VIDEO .MP4". Leading with the distinguishing noun makes the pair instantly differentiable and tightens width disparity. Keep one verb only if you collapse to a primary/secondary split. — `components/studio/video-stage.tsx`

## P2 — Lower priority / polish

### Encoder & pipeline correctness

- [ ] **VFR sources get wrong/overlapping output timestamps (constant duration vs real CTS)** — `timestamp = round((decoded.timeSec - safeIn)*1e6)` preserves real (possibly variable) presentation times, but every frame's duration is forced to a constant `usPerFrame = round(1e6/avgFps)`. For VFR sources, consecutive real timestamps can be closer than `usPerFrame` (overlapping durations) or farther (gaps). The `firstTimestampBehavior:'offset'` choice is sound for DTS; the latent bug is duration/timestamp mismatch on VFR. Harmless for CFR H.264 (the common case). — **Fix:** Derive duration from the delta to the next frame's timestamp (one-frame lookahead), or drop explicit duration and let timestamps carry timing. Otherwise detect VFR (timescale/delta variance) and warn, documenting that VFR inputs are normalized to avg-fps timing. — `lib/studio/video-export/encode-filtered-video.ts:226-235`

- [ ] **`isSupported` never called on the real target dims — unsupported dimensions surface as a late encoder error** — `ExportEncoder.isSupported()` exists but `encodeFilteredVideo` constructs the encoder directly with clamped/evened dims. The VideoStage probe (video-stage.tsx:121) only checks a fixed 1280x720 config, not the actual export dims. For an unusual aspect ratio the evened target could be HW-unsupported, and `configure()` throws asynchronously into `encoder.error` → `encoderError`, surfaced only at the next `add()`/`finish()`. (`makeEven` on a 1px source → 2 is fine.) — **Fix:** `await ExportEncoder.isSupported({width:targetW,height:targetH,fps})` inside `initPipeline` before constructing, throwing a clean "unsupported dimensions/codec" error so the UI shows a graceful message. — `lib/studio/video-export/encode-filtered-video.ts:54-57, 191-195`; `encoder.ts:90-101`

- [ ] **RenderCore.create has no abort/timeout — a stuck shader mount during init hangs the export** — If `initPipeline()` rejects, the real error does surface correctly (propagates out of `await pendingInit` → `consumerError` → orchestrator rethrow); the "no frames decoded" branch (line 261) is only hit on zero in-range frames, so that's fine. The gap: `RenderCore.create` awaits a single rAF + ref callback with no timeout and no `AbortSignal`, so a shader that never mounts hangs the export, and abort during the init window isn't honored until the first post-init `throwIfAborted`. — **Fix:** Pass the `AbortSignal` into `RenderCore.create` and reject its mount-wait promise on abort + a timeout. — `lib/studio/video-export/encode-filtered-video.ts:212-218, 261-263`

- [ ] **`ExportEncoder.dispose()` doesn't finalize the muxer or release `target.buffer`** — On abort/error the orchestrator calls `encoder.dispose()` (encode-filtered-video.ts:274). It closes the `VideoEncoder` (the resource that matters) but leaves the Muxer un-finalized and `target.buffer` (potentially many MB of partially-muxed data) referenced until the instance is GC'd. Not a true leak since the instance is dropped immediately, but no explicit eager release. — **Fix:** Optionally null out `this.muxer`/`this.target` in `dispose()` to drop the buffer eagerly on abort of a large export. — `lib/studio/video-export/encoder.ts:126-135`

### Frame source robustness

- [ ] **`detectFps` uses whole-track average — short ranges get a misleading `estimatedTotal` but no correctness bug** — `fps = nb_samples/durationSec` is the whole-clip average; fine as an encoder hint. `estimatedTotal` (rangeSec*fps) is only the progress denominator and is defended with `Math.max(estimatedTotal, framesDone)` (encode-filtered-video.ts:249), so progress never exceeds 100% or divides by zero. Progress can jump if avg fps differs from local fps. — **Fix:** None required. Optionally count in-range samples for a tighter progress estimate. — `lib/studio/video-export/frame-source.ts:87-94`

- [ ] **`getDecoderDescription` slices 8 bytes assuming the avcC/hvcC box is exactly the buffer — fragile but works** — `configBox.write(stream)` writes only that box, so slicing 8 bytes (size+fourcc) yields the config record — correct for avcC/hvcC. It does not validate box length vs buffer length. Codec error handling is already graceful: unsupported codec throws synchronously in `onReady` → rejects (216-218); unparseable container never fires `onReady` and the `if (!info)` check (253) throws cleanly after flush. HEVC input decodes fine but always re-encodes to hardcoded H.264 Baseline (intended). — **Fix:** None functional. Optionally assert `stream.buffer.byteLength > 8` before slicing for a clearer malformed-box error. — `lib/studio/video-export/frame-source.ts:58-77`

### Frame renderer perf/correctness

- [ ] **Per-frame PNG encode+decode roundtrip; readback gate checks content-presence not content-change** — Each frame does `toBlob('image/png')` → objectURL → `Image.decode()` → setUniforms → rAF readback → drawImage into a NEW output canvas (full PNG roundtrip per frame — slow but functional; blob URL is correctly revoked at line 226). The `sampler.hasContent(canvas, elapsed)` gate only checks non-blank, and with stable GL canvas identity + `preserveDrawingBuffer:true`, frame N+1's gate can pass immediately on frame N's still-present content — so a frame whose upload silently failed could be encoded as a duplicate of the previous frame with no error. — **Fix:** Skip the PNG roundtrip (reuse a single `ImageBitmap`→canvas→`createImageBitmap` path or cache one Image and swap `src`). Make the gate hash/sample a region and compare to the previous frame's signature to detect actual change, not just non-blank. Reuse one output canvas instead of allocating per frame. — `components/studio/video-export/frame-renderer.tsx:104-237`

### Studio wiring

- [ ] **Dynamic import in `runVideoExport` is unguarded — chunk-load failure is indistinguishable from a real encode failure** — `videoFileRef.current` is read once and captured by value (a mid-export video swap doesn't corrupt the run — good); `activeId`/`values` are closure-captured so a mid-export shader/param change keeps using the original (acceptable). Confirmed safe: aborted exports don't download a partial file because `encodeFilteredVideo` throws (AbortError) before `finish()`. Real gap: `await import(...)` (line 338) is unguarded — a chunk-load failure (offline, deploy mid-session) rejects into VideoStage's generic "Couldn't export the video" message, indistinguishable from a real encode failure. — **Fix:** Wrap the dynamic import in try/catch and surface a distinct "export module failed to load" message. No change needed for file/shader freshness. — `components/studio/studio.tsx:329-353`

### Accessibility polish

- [ ] **Per-frame progress on a polite live region is coalesced and mostly dropped** — Progress is announced via two `sr-only role="status" aria-live="polite"` regions (604-609) bound to `seqStatus`/`vidStatus`, updating every frame. Polite regions only announce when the SR is idle and coalesce rapid changes, so a fast 30-frame run speaks only a couple of intermediate values plus the final "Saved". The visible status spans (535, 549, 568, 582) are also not programmatically linked to the export/cancel buttons. — **Fix:** Throttle the announced value to milestones (~10% steps or stage changes: "Extracting frames", "Rendering", "Packaging", "Saved N frames"), keeping the visible per-frame text unthrottled. Keep `aria-live="polite"`. Optionally expose a progressbar with `aria-valuenow/min/max` and link the status text to its Cancel button via `aria-describedby`. — `components/studio/video-stage.tsx:604-609`

- [ ] **No `prefers-reduced-motion` handling (low risk here, unverified globally)** — The panel uses only `transition-colors` (no transform/opacity loops) and the video never autoplays (explicit comment line 378), so the export panel itself has minimal motion and progress is text, not an animated spinner — all good. But there is no `prefers-reduced-motion` media query on this surface, so any future/global animation (or the shader preview) could ignore the preference. — **Fix:** Add a `@media (prefers-reduced-motion: reduce)` rule (or Tailwind `motion-reduce:` variants) app-wide; keep export progress as text. Confirm the WebGL shader preview and any compare-slider animation honor the preference. — `components/studio/video-stage.tsx`

### Design polish

- [ ] **Single dense row crams range controls and both exports together** — On desktop everything sits in one row (panel container line 484, `sm:flex-row sm:justify-between`): Set in / in 0:00 / Set out / out 0:02 / Frames [8] on the left, both export buttons on the right. The left cluster mixes three interaction types (toggle buttons, readout spans, a number input) at the same `gap-2` rhythm with no internal grouping, so the eye can't separate "define the range" from "how many frames" from "export". — **Fix:** Group by task with spacing, not new chrome: tighten Set-in/readout and Set-out/readout into two paired units (`gap-1` within a pair, `gap-3` between), then a clear gap before Frames, then the export cluster pushed right. The Frames field logically belongs to the frame-export action — tuck it nearer that button rather than next to the out-point readout. — `components/studio/video-stage.tsx:484, 509-528`

- [ ] **Fallback line is contextually detached from the button it explains** — The "MP4 export needs Chrome, Edge, or Safari — frame zip works everywhere." line (598-602) renders as a flex sibling with `sm:basis-full`, wrapping full-width beneath both clusters. It only appears when `!videoExportOk` (the MP4 button is hidden in that state, 564-565), so the user sees a sentence about a button that isn't there, sitting below the still-present frames export. (See the P1 a11y item for the live-region/contrast fix on this same line.) — **Fix:** Anchor the fallback to where the MP4 button would have been (right cluster), styled as a quiet inline note ("MP4 unavailable here"), so it reads in the action's slot. Lead with the constraint, keep the reassurance second. — `components/studio/video-stage.tsx:598-602`

- [ ] **Running-state status text and idle button shift layout/weight; two parallel status blocks possible** — During export the button is swapped for a status span + Cancel (532-562 sequence, 564-595 MP4); running status uses `text-foreground` (full white) vs idle `muted-foreground` (535 vs 549). Two independent export blocks each able to show their own status/Cancel means up to two status strings + two Cancel buttons can coexist, and the swap changes cluster width, nudging neighbors. — **Fix:** Reserve a fixed-width status slot so swapping button↔status doesn't reflow neighbors. Since `anyRunning` already locks range controls (line 327) and each export disables the other (556, 589), use a single shared panel status line + one Cancel rather than two parallel ones — one job runs at a time. Keep the active/idle foreground-vs-muted distinction. — `components/studio/video-stage.tsx:532-595`

- [ ] **Saved-state checkmark glyph is the lone non-monochrome flourish** — Success states append a Unicode ✓ ("Saved 8 frames ✓" line 306, "Saved ✓" line 358). The glyph breaks the strict mono-type + shader-only-color identity, rendering with its own metrics and possible platform emoji styling. — **Fix:** Use a typographic mark consistent with the mono aesthetic (trailing "· saved" or a hairline SVG check in `currentColor`) so the only color in the UI remains the shader output. — `components/studio/video-stage.tsx:306, 358`
