## P0 — Blockers

- [ ] **Desktop stage has no transport / capture row** — Mobile shows a full transport (play, step, scrub, timecode, CAPTURE FRAME) but desktop reuses photo-flow chrome (IMAGE/CODE tabs, REPLACE) with no visible scrub or capture affordance; the two viewports tell different stories of the same feature and a desktop user has no obvious way to capture. — *Fix:* render the same transport component on desktop docked as one full-width bar under the canvas, matching mobile control order and the CAPTURE FRAME affordance; treat transport as one shared cross-breakpoint component and fix stage layout so it is always in view. — `docs/qa/audit/video-stage-desktop.png`

- [ ] **Capture draws before `seeked` — captured frame can be the wrong frame** — `seek()`/`step()` set `currentTime` synchronously and return without awaiting `seeked`; `capture()` pauses then immediately `drawImage`s, so an in-flight seek (fast scrub-then-capture, step+capture, or slow codec) paints the previously decoded frame. No guarantee `readyState >= HAVE_CURRENT_DATA` for the target time. — *Fix:* await a frame-ready signal before drawing — if `v.seeking`, await a one-shot `seeked` (with timeout fallback) and prefer `video.requestVideoFrameCallback()`; pass the exact target time into `captureVideoFrame` rather than re-reading `v.currentTime`. — `components/studio/video-stage.tsx:90-106`

- [ ] **Seek slider `step={0.01}` makes keyboard scrubbing effectively impossible (WCAG 2.1.1)** — Native arrow keys move one step = 10ms/press; traversing a 30s clip needs ~3000 presses, and PageUp/Down derive from the same step. The only practical scrub path is mouse drag (pointer-only). — *Fix:* decouple keyboard increment from capture precision — add `onKeyDown` so Arrow seeks ~1s, Shift/PageUp-Down ~10s, Home/End jump to 0/duration (call `seek()` + `preventDefault`); or raise `step` to ~0.1 / duration-100 and rely on frame-step buttons for sub-second precision. — `components/studio/video-stage.tsx:157-167`

## P1 — High

- [ ] **Video object URL leaks when switching video→photo (up to 200MB/session)** — On capture, mode stays `video` and `videoUrlRef` is intentionally kept for Recapture, but switching to photo mode and dropping a new image revokes the old image/frame and never revokes `videoUrlRef`; the prior video blob stays alive until unmount or a "New video" load. — *Fix:* revoke `videoUrlRef` and clear `videoUrl/videoName/videoDims` on mode change to photo (or inside `loadImageFile`/`commitImage` when abandoning a video for an image source); safe because the captured frame is its own blob. — `components/studio/studio.tsx:252-259`

- [ ] **`VideoStage` not keyed by src — stale duration/current/playing after Replace video** — A new `videoUrl` reuses the same component instance with no effect watching src, so `duration`/`current` retain the previous clip's values until `loadedmetadata` fires, and `initialTime` is only read at mount so a replace won't re-seek deterministically. — *Fix:* add `key={videoUrl}` to `<VideoStage>` so a new src fully remounts with clean state and re-applies `initialTime`. — `components/studio/studio.tsx:408-418`

- [ ] **Non-finite/late duration breaks slider and seeking** — `setDuration(v.duration || 0)` maps NaN→0 but leaves Infinity, giving the range input `max=Infinity` (unusable); when duration is 0 every `seek()` snaps to 0 so step/slider silently do nothing; no `durationchange` listener, so MOV/WebM duration updates after `loadedmetadata` never refresh the slider max or `onMeta`. — *Fix:* add an `onDurationChange` handler that re-reads duration, updates state, and re-notifies `onMeta`; if `!Number.isFinite(d)` disable slider/step and guard the `initialTime` seek with `Number.isFinite(v.duration)`. — `components/studio/video-stage.tsx:46-52, 159-167`

- [ ] **`timeupdate` overwrites slider value mid-drag, causing thumb stutter/jump** — `value={current}` is driven by `handleTimeUpdate` on every (~4Hz) tick during playback; dragging the slider sets `current` via `seek()` but if the video is still playing, `timeupdate` keeps overwriting it and the thumb jumps. — *Fix:* track an `isScrubbing` ref and stop syncing `current` from `timeupdate` while a drag is active (and/or pause on pointer-down). — `components/studio/video-stage.tsx:54-59, 119-120`

- [ ] **`playing` state can desync from the element** — `togglePlay` handles the play() promise rejection, but with no `play`/`pause` event listeners any pause/play not via `togglePlay` (media keys, PiP, browser autopause) leaves the button label wrong; `capture()`/`step()` set playing imperatively, which is duplicative. — *Fix:* make `onPlay`/`onPause` on the `<video>` the single source of truth (`setPlaying(true/false)`), drive `togglePlay` purely via `v.play()/v.pause()`, keep `.catch` to swallow the interrupted-play AbortError. — `components/studio/video-stage.tsx:61-70, 121`

- [ ] **Download PNG can export a generative canvas unrelated to the video** — `canDownload = !stageOpen && (!!image || shader.category === 'generative')`; in video mode with a generative shader and no captured frame, the generative branch can enable Download and `startExport` runs with `imageUrl` null, exporting a generative frame that ignores the video. — *Fix:* in video mode require `!!image` regardless of shader category (only allow the generative export branch when `mode === 'photo'`); make export intent explicit per mode. — `components/studio/studio.tsx:303, 277-293`

- [ ] **Tainted-canvas / CORS failure is misreported as a generic capture error** — `drawImage` on a cross-origin video without `crossOrigin='anonymous'` taints the canvas and `toBlob` throws `SecurityError`; the generic catch surfaces "Couldn't capture this frame — try another moment", misdirecting the user. Safe today (object URLs are same-origin) but the invariant is undocumented and the `<video>` never sets `crossOrigin`. — *Fix:* document/enforce same-origin-only, set `crossOrigin='anonymous'` when src is not a `blob:` URL, and in the catch distinguish `SecurityError` to surface a cross-origin-specific message. — `lib/studio/capture-frame.ts:26-31`

- [ ] **Capture busy/disabled state not conveyed to assistive tech (WCAG 4.1.3 / 1.3.1)** — While capturing, the button is `disabled` (removing it from tab order and losing keyboard focus context) and the text changes to "Capturing…" with no `aria-busy`; the interim state isn't announced and competes with the failure path in the same polite live region. — *Fix:* prefer `aria-disabled` + guarding the handler over the `disabled` attribute (or move focus to the captured result), add `aria-busy={busy}`, and announce "Capturing frame…" on start. — `components/studio/video-stage.tsx:173-180`; `components/studio/studio.tsx:471-473`

- [ ] **Inline error/notice text is hidden on mobile — no visible failure feedback (WCAG 1.4.1 / 1.3.1)** — Capture/load errors (e.g. "Couldn't capture this frame", "Unsupported video", "over 200 MB") surface only via the top-bar notice span, which is `hidden ... sm:inline`, so at 390w sighted users get no visible feedback; only screen-reader users hear it via the live region. — *Fix:* show the notice on mobile (toast or inline below the top bar) so the same message reaching the live region is visibly presented at all breakpoints. — `components/studio/studio.tsx:335-337`

## P2 — Polish / lower-risk

- [ ] **Seek slider touch/keyboard precision: 4px track fails 24px/44px target guidance (WCAG 2.5.5/2.5.8)** — The range input is `h-1` (4px) with an unstyled default thumb; on the 390w stage this is the primary scrub interaction and the effective hit area is essentially the thumb only, far below 44px iOS / 48px Material; the icon buttons at `size-9` (36px) are borderline. *(Merged A11y P2 + Responsive P1.)* — *Fix:* wrap the input in vertical padding (e.g. `py-2.5`) to give a ~44px touch zone while keeping the thin visual line, style `::-webkit-slider-thumb`/`::-moz-range-thumb` to ≥24×24px, and bump icon buttons toward 40-44px (`min-h-11` on the row). — `components/studio/video-stage.tsx:157-167`

- [ ] **Transport overcrowds and wraps awkwardly at 390px** — Fixed items (play + steps + timecode + Capture + gaps ≈ 328px) leave only ~38px for a `flex-1 min-w-[120px]` slider, so in the `flex-wrap` row the slider squeezes to its min and pushes timecode/Capture to a cramped second line; spacing has no rhythm and Capture butts against the timecode. *(Merged Design P2 + Responsive P1.)* — *Fix:* give the transport an intentional mobile layout — slider on its own full-width row, with row 1 = play/steps/timecode/Capture and row 2 = full-width slider (`basis-full sm:basis-auto` or a grid); establish a three-group rhythm `[icons] | [scrub+timecode] | [CAPTURE FRAME]` with Capture visually separated; use tabular figures for timecode. — `components/studio/video-stage.tsx:128-181`

- [ ] **Solid white scrub knob is the brightest non-shader element** — On the optical-lab identity, brightness peaks should come only from the shader output; the fully-filled solid-white knob becomes the highest-contrast point in the chrome and reads as a default OS range input. — *Fix:* demote to monochrome — hollow/ring or muted-gray knob at chrome value, played track in slightly lighter gray (not pure white), reserve pure white for active/hover only, keep the track hairline-thin. — `docs/qa/audit/video-stage-mobile.png`

- [ ] **Compare labels baked onto the shader output read templated and are inconsistent** — "WEBGL/ORIGINAL" and "VIDEO/IMAGE DITHERING" corner labels are stamped over the imagery, adding chrome noise to the one surface meant to hold color/texture; the processed-half label also differs across viewports (VIDEO DITHERING desktop vs IMAGE DITHERING mobile). — *Fix:* move labels into the thin chrome above/below each half (or a single caption under the split) as quiet monochrome microtype; standardize the processed-half term across viewports; if on-image is unavoidable, low-opacity hairline corner text only. — `docs/qa/audit/video-captured-desktop.png`

- [ ] **Empty video dropzone reads as a blank rectangle with no in-zone affordance** — `video-drop.png` is a flat dark rectangle with a single small glyph orphaned at bottom-center; instructional copy lives only in the hero, with no explicit "drop a video / browse" label or dashed target inside the stage. — *Fix:* add a centered monochrome line-icon plus one line of microtype (e.g. "Drop a video, or browse"), optionally a hairline inset border; remove/recenter the orphaned bottom glyph; keep it monochrome, no accent color. — `docs/qa/audit/video-drop.png`

- [ ] **Mode toggle + post-capture action set lack one predictable grammar** — Photo flow uses IMAGE/CODE/REPLACE, video stage uses PHOTO/VIDEO + REPLACE VIDEO, captured uses RECAPTURE / NEW VIDEO / COMPARE ON / DOWNLOAD PNG; labels and ordering shift between flows so video feels adjacent-but-different rather than a first-class sibling. — *Fix:* define one shared action grammar — fixed mode-toggle position, then a consistent source action (Replace photo/video), then a consistent output row (Compare toggle + Download PNG) — with identical verb casing/ordering so switching modes swaps only the noun. — `docs/qa/audit/video-captured-desktop.png`

- [ ] **Frame step assumes 30fps — skips/duplicates frames on 24/25/50/60fps content** — `FPS_ASSUMED=30` nudges `currentTime` by 33.3ms; 24fps can land in the same frame, 60fps skips every other; "frame" is approximate (and these buttons become the load-bearing accessible precision path once the P0 slider fix lands). — *Fix:* use `requestVideoFrameCallback` to advance exactly one presented frame where available, or derive fps from the first two rVFC timestamps; otherwise label the buttons as a ~33ms nudge rather than "frame". — `components/studio/video-stage.tsx:6, 80-88`

- [ ] **Mode toggle uses `aria-pressed` toggle-group instead of radiogroup (WCAG 4.1.2)** — Photo/Video is single-select (radio semantics) but renders `role="group"` with two `aria-pressed` buttons, announced as toggle buttons and not communicating the one-of-two relationship or supporting arrow-key roving. (The Compare on/off `aria-pressed` is correct — genuine toggle.) — *Fix:* convert `ModeToggle` to `role="radiogroup"` with `role="radio"` + `aria-checked` and roving tabindex/arrow keys; at minimum keep the group label. — `components/studio/studio.tsx:506-531`

- [ ] **`loadSampleImage` has no `onerror` — a failed sample load silently does nothing** — Only `img.onload` is set; a load failure leaves a dead click with no notice. Low impact (bundled asset). — *Fix:* add `img.onerror` to `flashNotice`, mirroring `loadImageFile`. — `components/studio/studio.tsx:490-504`

- [ ] **Paste handler ingests media without `preventDefault` and may double-handle** — `onPaste` uses the first image/video clipboard item without `e.preventDefault()`, so a paste into a focused editable element elsewhere is also handled. Low risk in this single-canvas studio. — *Fix:* call `e.preventDefault()` when a media item is ingested and ignore paste when the event target is editable. — `components/studio/studio.tsx:206-212`

- **Captions 1.2.2** — Preview is always muted with no `<track>`; with no audio ever rendered, 1.2.2 is reasonably N/A. Document the muted-preview rationale to justify the lint suppression; add a caption track only if audio playback is ever introduced. (`components/studio/video-stage.tsx:111-124`)

(`components/studio/studio.tsx:319-321`)
