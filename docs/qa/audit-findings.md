# Aperture Studio ‚Äî QA Punch List

**Build:** PASS ‚Äî `tsc --noEmit` clean (exit 0), `next build` clean (Next.js 16.2.9 / Turbopack, ~1.2s, 4/4 static pages, no warnings/errors). No files were edited during the audit.

**Responsive verdict:** Functional at all widths but the **tablet (md, ~768‚Äì1023px) two-pane layout is the weak point** ‚Äî a fixed-height canvas pane against a narrow rail produces a large vertical letterbox that reads as unintentional dead space. Desktop (1280) and mobile (390) are largely sound; tablet needs the structural fix below.

---

## P0 ‚Äî Must fix before ship

- [ ] **Tablet canvas letterbox ‚Äî the "dead space" problem** ‚Äî On md+ the canvas pane is locked to `clamp(360px, 52vh, 520px)` while the fitted 1.6-AR image only needs ~294px tall in the ~470px-wide canvas, leaving 70‚Äì220px of black `#070809` banding above/below the photo. The rail inherits the same too-tall, sparsely-filled column. Single most unintentional-looking state in the set. ‚Äî *Fix:* Stop driving md pane height from a fixed vh; size the pane to the fitted image. Best path is the P1 layout change below (stack at md, two-pane only at lg), which removes the narrow-canvas letterbox entirely. If keeping two-pane at md, derive height from width: `isDesktop ? 'clamp(440px, 72vh, 760px)' : 'min(60vw, clamp(320px, 46vh, 460px))'` and lower the floor so short images don't over-letterbox. ‚Äî `components/studio/studio.tsx` (lines 75‚Äì77)

- [ ] **Rail header shows leftover template/component-kit labels** ‚Äî Above "TRANSFORM YOUR PHOTO" the rail renders `IMAGE FILTERS 8   LOGO 3   GENERATIVE 28`. "LOGO 3" and "GENERATIVE 28" are meaningless to users and read exactly like un-renamed UI-kit instance labels ‚Äî the most AI/templated-looking artifact in the product. Also jarring on mobile. ‚Äî *Fix:* Delete "LOGO 3" and "GENERATIVE 28" entirely (on every breakpoint). If a section label is wanted, keep one honest eyebrow like "FILTERS" or "8 FILTERS" in muted mono uppercase and let "Transform your photo" be the only heading. Never ship enumerated component-kit labels. ‚Äî `components/studio` rail header component

- [ ] **Filled white DOWNLOAD PNG button out-shouts the shader** ‚Äî The thesis is "color only from the shader; chrome stays monochrome and quiet," but the solid chalk-filled `DOWNLOAD PNG` button (top-right, also in empty + mobile states) is pure `#e8ebf0` at full value and pulls the eye off the rendered output ‚Äî the opposite of an optical lab where the image is the hero. COMPARE ON is correctly ghosted; the two mismatch in weight. ‚Äî *Fix:* Demote DOWNLOAD PNG to the same ghosted/outline treatment as COMPARE (hairline border, `--foreground` text, transparent fill), or reserve a single filled button only for the primary moment / on-hover after a render. One filled control max, never brighter than the shader output. ‚Äî `components/studio` download button component

- [ ] **Export caps below requested resolution ‚Üí soft/blurry PNG** ‚Äî `maxPixelCount = width*height` is a budget on canvas pixels `(cssW*cssH*pixelRatio¬≤)`. With `minPixelRatio:1` and the budget set exactly to `width*height`, internal DPR ceil/rounding and `fit:'cover'` supersampling frequently push the GL canvas just over budget so it's scaled *down*; `ctx.drawImage(canvas, 0,0,width,height)` then upscales a smaller buffer to a soft PNG that doesn't match the preview. The content gate passes (non-blank), so it ships silently. ‚Äî *Fix:* Set `maxPixelCount` generously above target (e.g. `width*height*4`, or `Number.MAX_SAFE_INTEGER` and rely on `clampToMaxSide` for the hard cap), pin both `minPixelRatio` and `maxPixelRatio` to 1 so the off-screen buffer is exactly `width√óheight`, and assert `mount.canvasElement.width === width` after the gate (redraw at true ratio if not). ‚Äî `components/studio/export-renderer.tsx` (line 159)

- [ ] **`useMediaQuery` starts false ‚Üí guaranteed desktop layout flash + extra GL resize on every load** ‚Äî `isWide`/`isDesktop` init to false and only flip after the mount effect, so server and first client paint render the mobile branch, then a second paint snaps to the desktop pane height (clamp 440‚Äì760px). The studio visibly jumps height on every desktop load/refresh ‚Äî the exact hydration flash the comment claims to avoid ‚Äî and the resize forces PreviewBox's ResizeObserver to recompute and the WebGL canvas to resize. ‚Äî *Fix:* Read `matchMedia` synchronously in a guarded `useState` initializer: `useState(() => typeof window !== 'undefined' && window.matchMedia(query).matches)` so the first client paint is correct. Alternatively gate the studio body behind a `mounted` flag rendering a fixed-height skeleton so server and first client paint agree. ‚Äî `components/studio/studio.tsx` (line 33)

- [ ] **Randomize/Reset can drop params the shader needs (latent correctness trap)** ‚Äî `randomize()`/`reset()` call `onReplaceValues` with an object containing only the keys they iterate (range/boolean/enum/color/palette). Any param whose control type isn't one of those is silently omitted, and the shader falls back to library defaults mid-session ‚Äî a visible jump distinct from the deliberate randomization. Low blast radius today (all current controls covered) but breaks the moment a new control type is added. ‚Äî *Fix:* Seed `next` from a full clone of current values or `initialValues(shader)` and only overwrite the keys randomize varies: `const next = { ...initialValues(shader) }; for (const p of shader.params) { ... }`. Add an exhaustive default branch (or a `never` check) so an unhandled control type keeps its current value. ‚Äî `components/studio/control-panel.tsx` (line 36)
